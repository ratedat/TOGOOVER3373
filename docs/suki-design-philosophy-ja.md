# Suki UI デザイン思想

この文書は、RHODES OBS COMMANDER3373 の Suki/Avalonia UI を今後拡張していく時の判断基準をまとめる。見た目の好みではなく、オペレーター、秘宝、統合戦略ごとの特殊値、OCR候補、OBS表示、ADB/MAA/GLM/Ollama などが増えても破綻しないための設計思想を定義する。

## 基本方針

Suki UI は、一般ユーザー向けのランディングページでも、装飾的なダッシュボードでもない。目的は、統合戦略の現在状態を正確に確認し、OCRやテンプレート認識の結果を検証し、必要な修正を素早く行い、OBSやサイドカーへ安定して表示するための実務用ワークベンチである。

このため、UIは「見栄え」よりも「繰り返し確認しやすいこと」「状態の出所が分かること」「誤認識を直せること」「要素が増えても迷わないこと」を優先する。

## 1. ラン状態を常に中心に置く

すべての操作は現在のラン状態に影響する。キャンペーン、分隊、等級、希望、源石錐、耐久、シールド、指揮レベル、IS固有値などは、他の画面に移っても文脈として失われてはいけない。

そのため、上部には常にランの文脈を置く。ここは詳細編集の場所ではなく、現在どのランを編集しているか、どの端末やOCRエンジンで取得しているかを確認するための場所にする。

## 2. データ種別ではなく作業目的で分ける

UIの大分類は、単に「秘宝」「オペレーター」「設定」を並べるのではなく、ユーザーが何をしたいかで分ける。

- `Run`: ラン基本値とIS固有値を編集する。
- `Choices`: オペレーター、秘宝、将来追加される選択式カタログを扱う。
- `Recognition`: OCR、テンプレート認識、候補レビュー、ログ、スクリーンショットを扱う。
- `Output`: OBS、サイドカー、大会向け表示を扱う。
- `Runtime`: ADB、MAAFramework、GLM-OCR、Ollama、Hyper-Vなどの外部実行環境を扱う。
- `Debug`: ログ、トレース、バグ報告、検証用プローブを扱う。

新しい項目が増えても、まずこのどこに入るかを判断する。新しいトップレベルタブは、既存の作業目的に入らない時だけ追加する。

## 3. 状態編集と認識レビューを分離する

OCRやテンプレート認識の結果は、ゲーム状態そのものではない。認識結果は「候補」であり、状態へ反映する前に、必要なら確認や拒否ができる必要がある。

自動反映は便利だが、すべてを無条件に状態へ流し込むと、誤認識の原因が見えなくなる。候補には、認識元、raw text、正規化後の値、信頼度、スクリーンショットやcrop、detail JSON を紐づける。

## 4. 証拠を操作の近くに置く

デバッガーが必要としているのは、単なる成功/失敗表示ではない。なぜその値になったのか、どの画像から読んだのか、どのタスクが何を返したのかを追える必要がある。

そのため、右側にはインスペクタまたは証拠ペインを置く。選択中のオペレーター、秘宝、特殊値、OCR候補、ランタイム状態に応じて、詳細情報、ログ、認識証拠、適用/拒否操作を表示する。

## 5. 大量カタログはカード展示にしない

オペレーターや秘宝は数が多く、今後も増える。これらは見た目のカード一覧ではなく、検索、フィルター、選択状態、除外状態、優先表示、選択のみ表示を備えた作業用リストとして扱う。

一覧行は短く保ち、詳細はインスペクタへ逃がす。行の中に効果文、認識情報、デバッグ情報を詰め込みすぎない。

## 6. 増える要素はスキーマとレジストリで受ける

今後、キャンペーン固有値、認識プロファイル、表示ブロック、ランタイム機能、デバッグ項目は増える。これを画面ごとのハードコードで受けると、UIがすぐに壊れる。

新しい要素は次のメタデータを持つべきである。

- 安定したID
- 表示名
- 所属ワークスペース
- 状態の保存先
- 出所: 手入力、OCR、テンプレート、GLM、インポート、ランタイム検出など
- 表示優先度
- インスペクタ表示
- 空状態とエラー状態の文言
- OBS/サイドカーへ出すかどうか
- 自動反映できるか、レビュー必須か

実装上は、ランフィールド、選択カタログ、認識結果、出力パーツ、ランタイム機能をそれぞれレジストリ化する。呼び出し側が個別キャンペーンや個別フィールドを知りすぎない形を目指す。

## 7. MAAFramework の語彙に寄せる

このアプリは MAAFramework ファミリーとして進める方針なので、認識タスク、プロファイル、リソース、テンプレート、結果の粒度は可能な限り MAA 側の考え方に寄せる。

独自UIの都合で概念名を増やしすぎると、MAAの知見を取り込みにくくなる。UI上の名称と内部のタスク名は、完全一致できない場合でも対応関係が追えるようにする。

## 8. 重い機能は任意導入として明確に扱う

GLM-OCR や Ollama は精度向上に有効だが、全ユーザーに必須として押し付けるものではない。UI上では、未導入、導入済み、起動中、モデル未取得、失敗、アンインストール可能といった状態を明確に分ける。

任意機能であるほど、設定場所を隠してはいけない。Runtime ワークスペースに集約し、OBSや選択編集の画面に混ぜない。

## 9. OBS表示とデータ編集を混ぜない

OBSやサイドカーは、現在状態をどう見せるかの出力層である。ラン状態やOCR候補の編集とは責務が違う。

出力設定では、表示パーツ、スクロール、表示除外、大会向け表示、別ウィンドウ化を扱う。一方で、値そのものの編集や認識候補の採否は Run、Choices、Recognition 側で行う。

## 10. 視覚表現は静かで密度の高いものにする

このUIは長時間の検証に使う。装飾の強いグラデーション、大きなヒーロー、余白の多いカード型ダッシュボードは向かない。

基本は暗色、低い彩度、薄い境界線、固定幅の列、安定した行高、読みやすいログ表示とする。テキスト、パス、JSON、効果文が長くなっても、レイアウト全体が押し広げられないようにする。

## 実装で迷った時の判断基準

新しいUIを追加する前に、次を確認する。

- これはラン状態を編集するものか、認識結果を確認するものか、出力を変えるものか、ランタイムを管理するものか。
- 既存の `Run`、`Choices`、`Recognition`、`Output`、`Runtime`、`Debug` のどこに入るか。
- 一覧行に出すべき最小情報は何か。
- 詳細と証拠はインスペクタへ逃がせるか。
- 自動反映してよいか、レビューが必要か。
- OBS/サイドカーへ出す必要があるか。
- 将来同じ形の要素が増えた時、同じ仕組みで追加できるか。

この問いに答えられないUIは、まだ実装に入らない方がよい。

## まとめ

Suki UI の設計思想は、拡張可能な実務用ワークベンチである。重要なのは、すべてを1画面に詰め込むことではなく、ラン状態、選択、認識、出力、ランタイム、デバッグを明確に分け、それぞれを証拠付きで編集できるようにすること。

今後要素が増えても、トップレベルタブを増やし続けるのではなく、スキーマ、レジストリ、インスペクタ、レビューキューで受け止める。これにより、OCR精度改善、MAAFramework連携、OBS表示、検証ツールとしての拡張を同時に進められるUIにする。
